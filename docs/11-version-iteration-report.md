# 版本迭代文档

## 1. 版本迭代背景

《藻井集》当前已经从早期项目骨架推进到一个可运行、可演示的文化主题交互网站原型。根据 `README.md`、`docs/00-project-brief.md`、`docs/01-product-requirements.md` 与 `docs/02-information-architecture.md`，项目目标是围绕“仰望 — 解构 — 读纹 — 拾色 — 再造”的路径，把藻井图像组织成可浏览、可理解、可用于设计转译的视觉资源系统。

初版已经完成的基础能力包括：

- 基于 React + Vite + TypeScript + React Router 的前端项目结构。
- 4 个主路由：`/`、`/atlas`、`/atlas/:dynastyId`、`/work/:workId`，对应 `src/App.tsx`。
- 首页沉浸式入口与加载动画，当前实现包含 `IntroLoading` 与 `IntroVideo`。
- `/atlas` 总览状态与朝代索引状态，当前由 `AtlasPage.tsx` 的 `atlasView = "overview" | "dynasty-index"` 控制。
- 13 个朝代条目，来自 `src/data/dynasties.ts`。
- 51 个作品条目，来自 `src/data/works.ts`。
- 图鉴环、朝代卡片、选中朝代作品浏览和作品详情入口。
- 三兔飞天藻井详情展示页面，包括时代背景、核心纹样形态和色卡/生成工具相关页面组件。
- 基础构建脚本：`npm run check`、`npm run build`。

本轮迭代的直接触发来自两类问题：

1. 项目从 `zaojingji-complete` 合并 Gemini 更新后，需要确认加载动画、icon、具体页面和资源路径能够在本地项目中稳定运行。
2. 用户打开图鉴页后发现环形图鉴图片破图，需要从代码、资源路径和浏览器状态三层定位问题。

本轮迭代的核心目标不是单纯增加页面，而是围绕以下问题优化产品表达：

- 用户首次进入时是否能理解“藻井集”的定位。
- 用户能否从首页进入图鉴，再进入朝代和作品详情。
- 色卡功能是否从装饰性展示转向有工具价值的可复制、可调整、可导出功能。
- 藻井内容是否从静态介绍转向可探索的朝代、作品、结构和色彩组织。
- 当前 MVP 是否适合作为课程答辩和 AI 产品经理面试佐证项目。

## 4. 关键迭代点说明

### 迭代点 1：首页定位与进入路径

- 原始问题：初版首页已规划“抬头观看”体验，但合并 Gemini 更新后需要确认 loading、intro 视频和进入路径是否真实可用。
- 用户反馈/测试证据：用户要求重点更新加载页动画；本轮检查发现 `IntroLoading.tsx` 引用的 `/zaojing_loading.mp4` 不存在。
- 产品判断：首页是项目定位的第一触点。如果 loading 动画缺失或报错，用户无法顺利建立“藻井空间入口”的第一印象，优先级高于后续高级功能。
- 修改方案：
  - 同步 `IntroLoading.tsx`、`IntroVideo.tsx`。
  - 将真实 loading 视频复制到 `public/assets/loding/loading.mp4`。
  - 将视频引用改为 `/assets/loding/loading.mp4`。
  - 修正 `index.html` 标题和 favicon。
- 涉及页面/文件：
  - `src/routes/IntroPage.tsx`
  - `src/components/intro/IntroLoading.tsx`
  - `src/components/intro/IntroVideo.tsx`
  - `public/assets/loding/loading.mp4`
  - `index.html`
- 修改后效果：首页能加载 loading 视频，随后进入 intro/main 阶段；Playwright 验证 `loading_video_src` 为 `/assets/loding/loading.mp4`。
- 当前状态：已完成。

### 迭代点 2：藻井图鉴浏览

- 原始问题：图鉴总览是核心浏览入口，用户截图中环形卡片图片显示破图。
- 用户反馈/测试证据：用户提供 `/atlas` 截图，说明所有图片路径都在 `public/assets/zaojing/originals`。
- 产品判断：图鉴页是用户从“看见藻井”进入“浏览藻井”的关键路径。图片破图会直接破坏项目图鉴属性，优先级高。
- 修改方案：
  - 检查 `AtlasMorphRing.tsx` 确认使用 `image.imagePath`。
  - 检查 `overviewImages.ts` 确认图片来自 `works.images.thumbnail`。
  - 检查 `works.ts` 确认 base path 为 `/assets/zaojing/originals`。
  - 直接请求图片 URL，确认本地服务返回 200。
  - 用 Playwright 检查图片 `naturalWidth > 0`。
  - 给图片 URL 加版本参数 `?v=20260629`，解决浏览器旧 404 缓存问题。
- 涉及页面/文件：
  - `src/components/atlas/AtlasMorphRing.tsx`
  - `src/data/overviewImages.ts`
  - `src/data/works.ts`
  - `public/assets/zaojing/originals/*.png`
- 修改后效果：当前服务截图 `output/atlas-check.png` 中图鉴图片可正常显示；Playwright 检查无本地 404。
- 当前状态：已完成。

### 迭代点 3：朝代/分类结构

- 原始问题：初版文档规划从 6 件重点作品起步，后续需要形成更完整的朝代图鉴。
- 用户反馈/测试证据：已有开发日志和代码显示项目已从 6 个重点朝代扩展为 13 个朝代；当前 `src/data/dynasties.ts` 有 13 个 `id` 条目。
- 产品判断：朝代结构决定用户能否理解藻井在历史阶段中的差异，是文化内容组织的主线。相比收藏、导出等高级功能，朝代结构更影响首次理解和主路径完整度。
- 修改方案：
  - `/atlas` 中 `dynasty-index` 状态按 `dynasty.order` 渲染朝代卡片。
  - `/atlas/:dynastyId` 读取路由参数并筛选对应作品。
  - 设置上一朝代/下一朝代导航能力。
- 涉及页面/文件：
  - `src/routes/AtlasPage.tsx`
  - `src/routes/SelectedDynastyPage.tsx`
  - `src/data/dynasties.ts`
  - `src/utils/getDynastyById.ts`
  - `src/utils/getWorksByDynasty.ts`
- 修改后效果：用户可从总览进入朝代索引，再进入具体朝代作品浏览。
- 当前状态：已完成基础结构，滚动末端鼠标点选等增强交互未系统验证。

### 迭代点 4：藻井详情页

- 原始问题：PRD 和页面规格希望详情页包含概览、投射、色彩、结构、生成五个模块，并最好通过 Tab 或内部状态切换。
- 用户反馈/测试证据：用户提到 Gemini 主要更新“最后具体页面的三个页面”；当前 `WorkPage.tsx` 已接入 `Page11`、`Page18`、`PageArtGenerator`。
- 产品判断：详情页是将图鉴浏览转化为“文化解释 + 设计转译”的关键页面。当前优先保证答辩展示完整度，而不是一次性完成全量数据驱动详情架构。
- 修改方案：
  - 将远端具体页面组件同步到本地。
  - `WorkPage.tsx` 使用缩放容器展示 1920px 宽的详情页面组合。
  - 接入时代背景、核心纹样形态、色卡/生成工具三个视觉页面。
- 涉及页面/文件：
  - `src/routes/WorkPage.tsx`
  - `src/components/work/Page11.tsx`
  - `src/components/work/Page18.tsx`
  - `src/components/work/PageArtGenerator.tsx`
- 修改后效果：用户可进入作品详情展示页，看到三兔飞天藻井的时代背景、结构拆解和色卡生成工具。
- 当前状态：部分完成。当前详情页偏 Figma 展示页，尚未完全实现 `docs/04-page-spec.md` 中的数据驱动 Tab 结构。

### 迭代点 5：色卡提取/色彩分析

- 原始问题：项目核心创新点之一是将藻井色彩从审美展示转化为可提取、可应用的信息。
- 用户反馈/测试证据：README、PRD 和视觉系统均强调“文化语义色卡”和“设计转译”；当前 `PageArtGenerator.tsx` 包含 Canvas 采样取色、色卡显示、HEX 复制、手动调色、重新生成和 PNG 下载。
- 产品判断：色卡功能影响项目创新点表达，优先级高。但如果做完整文化语义取色系统，内容和算法成本较高，因此当前先保证可演示的工具行为。
- 修改方案：
  - 保留 `PageArtGenerator` 中的默认色板。
  - 支持上传图片后用 Canvas 采样生成 8 个颜色。
  - 支持点击复制 HEX。
  - 支持手动调整颜色。
  - 支持根据色卡生成图案化艺术画布并下载 PNG。
- 涉及页面/文件：
  - `src/components/work/PageArtGenerator.tsx`
- 修改后效果：色卡从静态装饰变成有操作反馈的工具模块。
- 当前状态：部分完成。当前实现具备色彩提取和生成工具能力，但 `works.ts` 中 `palettes`、`colorHotspots` 仍为空数组，文化语义色卡尚未数据化。

### 迭代点 6：文化解释与内容层级

- 原始问题：项目不能只展示图片，需要解释朝代背景、纹样结构、文化寓意和设计价值。
- 用户反馈/测试证据：`docs/00-project-brief.md`、`docs/01-product-requirements.md` 和 `docs/04-page-spec.md` 都要求文化解释；当前 `Page11.tsx` 包含“时代背景”“核心图案”“创新与价值”，`Page18.tsx` 包含“核心纹样形态”和结构说明卡片。
- 产品判断：文化解释是项目区别于普通图片库的关键。但历史、洞窟、纹样寓意必须校验来源，不能把 AI 生成解释当作最终学术事实。
- 修改方案：
  - 在展示页加入时代背景、核心图案、创新价值和结构说明。
  - 在 `works.ts` 中保留 `sourceRefs`、`catalogNos` 和 `verificationStatus: "needs-verification"`。
- 涉及页面/文件：
  - `src/components/work/Page11.tsx`
  - `src/components/work/Page18.tsx`
  - `src/data/works.ts`
  - `docs/10-reference-materials.md`
- 修改后效果：详情页不再只是图像展示，而是开始组织“背景 — 结构 — 价值”的内容层级。
- 当前状态：部分完成。文化解释已有展示，但学术来源、逐条注释和多作品内容仍待补充。

### 迭代点 7：视觉系统与交互反馈

- 原始问题：文化主题网页容易变成普通资料页或过度国潮海报，项目需要保持暗场、图鉴、解构、工具感。
- 用户反馈/测试证据：`docs/06-visual-system.md` 明确视觉关键词为“暗场、仰视、悬浮、图鉴、解构、取色、转译、克制、文化感、设计工具感”；当前页面使用黑底、暗金、米白、环形卡片、motion 动效。
- 产品判断：视觉统一直接影响课程答辩和作品集展示可信度。与新增高级功能相比，视觉一致性更影响用户首次理解。
- 修改方案：
  - 同步远端 `src/styles`。
  - 使用 `motion/react` 保留加载、图鉴和详情页动效。
  - 修改 favicon 和页面 title。
  - 修复图鉴图片缓存问题，避免视觉核心区域破图。
- 涉及页面/文件：
  - `src/styles/*.css`
  - `src/components/atlas/AtlasMorphRing.tsx`
  - `src/components/layout/SiteShell.tsx`
  - `index.html`
- 修改后效果：页面视觉更统一，顶部导航、图鉴环、详情页都形成暗场展示风格。
- 当前状态：已完成基础统一，移动端与细节截图证据待补。

### 迭代点 8：数据结构与后续可扩展性

- 原始问题：如果作品、朝代、图片和说明写死在组件中，后续扩展会困难。
- 用户反馈/测试证据：README 和 `docs/05-data-schema.md` 明确要求“所有藻井内容必须从 `src/data/` 中读取”；当前朝代、作品和总览图均来自 data 层。
- 产品判断：数据结构决定项目是否能从课程原型扩展为完整图鉴系统。优先维护数据层，比临时增加页面更有长期价值。
- 修改方案：
  - `dynasties.ts` 维护朝代数据。
  - `works.ts` 维护作品数据、图片路径、标签、来源和待校验状态。
  - `overviewImages.ts` 从作品数据中抽取总览图。
  - `getDynastyById`、`getWorksByDynasty`、`getWorkById` 提供读取工具。
- 涉及页面/文件：
  - `src/data/dynasties.ts`
  - `src/data/works.ts`
  - `src/data/overviewImages.ts`
  - `src/utils/*.ts`
- 修改后效果：朝代图鉴和作品图鉴具备继续扩展内容的基础。
- 当前状态：已完成基础数据层；色卡、纹样、基因卡的结构字段存在但内容大多待补。

## 5. 修改前后对照

### 对照 01：首页引导优化

- 修改前截图：`docs/evidence/screenshots/v0.1-home-before.png`
- 修改后截图：`docs/evidence/screenshots/v0.2-home-after.png`
- 修改前问题：首页入口依赖早期实现，Gemini 更新后的 loading 视频路径未正确落地。
- 修改后变化：补齐 `public/assets/loding/loading.mp4`，修正 `IntroLoading.tsx` 视频路径，页面 title 和 favicon 完成项目化。
- 迭代理由：首页是核心定位入口，优先保证首屏体验和资源稳定。

- 先解决核心路径：用户必须能从首页进入图鉴、朝代、作品详情。

- 暂缓高成本增强：收藏、导出完整卡片、对比、AI 建议、摄像头识别都可以提高上限，但不是当前 MVP 成败点。

## 10. 面试项目复盘提炼

### 10.1 项目一句话介绍

《藻井集》是一个将敦煌藻井图像结构化浏览、取色分析并转化为设计灵感的交互图鉴原型。

### 10.2 项目中的产品问题

这个项目解决的不是“做一个文化网页”，而是三个产品问题：

1. 如何让传统文化图像被结构化浏览  
   藻井资料本身分散，普通用户很难按朝代、作品和纹样结构理解它们。项目通过总览、朝代图鉴、单件详情的路径，把文化图像组织成可探索的信息架构。

2. 如何把藻井色彩从视觉审美转化为可理解、可提取、可应用的信息  
   色彩不只是“好看”，还需要被设计师理解和复用。当前版本通过色卡、HEX、复制、调色和生成画布，把色彩转为更接近工具化的信息。

3. 如何通过交互路径降低用户理解门槛  
   项目先用 loading 和沉浸式入口建立“抬头观看”的体验，再进入图鉴浏览，最后进入详情解构，避免一开始就堆叠文化说明。

### 10.3 我的角色

我在这个项目中主要负责产品方向拆解、信息架构设计、页面路径规划、反馈整理和版本迭代推进。

我先把项目目标拆成“仰望、解构、读纹、拾色、再造”五个阶段，再对应到首页、图鉴、朝代、详情、色卡和生成模块。开发过程中，我没有把所有想法一次性做完，而是先保证用户可以走通核心路径，再根据反馈优先修复 loading 视频、图鉴破图、页面标题和资源路径等会影响展示完成度的问题。

在收集反馈上，我主要通过打开本地页面、查看用户截图、检查构建和浏览器请求来判断问题优先级。例如图鉴页图片破图时，我没有直接改样式，而是先确认代码路径、静态资源请求和浏览器加载状态，最后判断是旧缓存问题，并通过版本参数解决。

这轮迭代让我把项目从“能跑的原型”推进到“更适合答辩和作品集展示的版本”，同时也明确保留了待开发边界，例如文化语义色卡、完整数据化详情页和移动端验证。

### 10.4 可用于简历的项目 bullet

- 负责《藻井集》文化交互网站的信息架构与版本迭代，将“仰望 — 解构 — 读纹 — 拾色 — 再造”拆解为首页、图鉴、朝代、详情、色卡与生成模块，形成可演示的核心用户路径。
- 基于小样本测试和用户截图反馈，定位并修复 loading 视频缺失、图鉴图片破图、模板 title/favicon 残留等影响展示完成度的问题，推动项目从原型状态迭代到答辩展示版。
- 设计并落地藻井图鉴的数据组织方式，使用 `dynasties`、`works`、`overviewImages` 等数据层承载 13 个朝代与 51 个作品条目，为后续文化内容扩展和详情页数据化重构建立基础。
- 推进色卡功能从静态展示转向工具化表达，接入图片取色、HEX 复制、手动调色和 PNG 下载等交互能力，并明确后续文化语义色卡、色彩对比和导出能力的产品路线。

### 10.5 面试讲述版本

《藻井集》是我做的一个敦煌藻井文化交互网站原型。它的核心不是简单展示敦煌图片，而是把藻井这种传统顶部装饰转化成一个可以浏览、理解、取色和再应用的设计资源系统。

我最开始把用户路径拆成五步：仰望、解构、读纹、拾色、再造。用户先通过沉浸式首页进入，再按朝代浏览藻井图鉴，选择具体作品后进入详情页，看时代背景、纹样结构和色彩提取，最后生成可用于设计参考的视觉结果。

在迭代过程中，我比较关注用户能不能理解路径和功能价值。比如用户打开图鉴页后发现图片破图，我没有只从 UI 层改，而是检查了数据路径、静态资源、浏览器请求和截图结果，最后判断是旧缓存导致，并通过图片 URL 加版本参数解决。另一个例子是 loading 动画，用户指出视频在具体目录，我就把资源补到 public 并修正组件路径，保证首页入口稳定。

目前这个项目已经具备首页、图鉴、朝代、详情和色卡生成工具的主路径，但我也没有把它包装成完全完成的产品。比如文化语义色卡、完整数据化详情页、移动端验证和学术来源校验还在下一版计划里。对我来说，这个项目更重要的是展示我如何根据项目目标、用户反馈和实现成本做产品判断，并把一个文化主题原型逐步迭代成可以答辩和面试讲述的产品案例。
